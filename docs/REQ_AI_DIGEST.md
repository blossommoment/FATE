# 需求文档：AI 总结层（人话成品）与黑话清扫

> 状态：待用户过目（文末 5 个待拍板问题）
> 日期：2026-07-03
> 一句话：**报告降级为账本，AI 总结升级为成品** —— 用户看的是 AI 给的人话标签
> 和生活建议，报告只在想查账时才展开。

---

## 一、产品重定位

```
现在：  计算器 → 报告（用户硬着头皮看，看不懂「食神温润」）
改后：  计算器 → 事实清单(账本) → AI 消化 → 人话成品（标签+建议+总结）
                     ↑ 想查依据才点开             ↑ 用户真正消费的东西
```

三条铁律（沿用 REPORT_STANDARD 的架构，只是消费端换了）：

1. **AI 的输入是计算器的结论，不是八字原文**——八字只是计算器内部的推导
   材料。AI 拿到的是：强弱、喜忌、四维、十神结构分、深维分数、依恋类型、
   岁运补耗判定。它做的是「把体检报告翻译成生活方案」，不是算命。
2. **标签和候选池由规则表定，AI 只做挑选与措辞**——保证同一个人任何时候
   生成的标签一致（确定性承诺不破），AI 的自由度只在组织语言。
3. **每盘只生成一次，缓存复用**——按 profile id 缓存，控制成本，也保证
   用户两次打开看到同样的内容。

## 四、确定性保障（不破既有承诺）

```
标签：规则表映射（计算器分数 → 标签），100% 确定，进黄金测试
候选池：规则表映射（喜忌×十神 → 行业/环境池），确定
AI 职责：从池中挑选、排序、成文 —— temperature 0.4，同输入基本同输出
缓存：profile id → 生成一次存储复用（本机 localStorage + 可选服务端）
兜底：AI 不可用时，展示规则表标签 + 模板句（无 AI 也不开天窗）
校验：AI 返回后正则抽标签与数字，与清单比对，不一致重试（沿用现有方案）
```

## 六、黑话清扫（并行工作流）

原则：**用户可见正文零命理黑话；术语一律降级到「依据」层**（点开才看到）。

- 已定位的典型：「食神温润，讷于言争」（合盘两人底色 trait 标题位）、
  「比劫立身，界限分明」同类判词、各处 basis 里的「食伤权重」类表述——
  正文位的全部替换为人话标签（如「温和的慢表达者」），原判词挪进依据折叠。
- 排查方式：全量扫描用户可见字符串，按「普通人测试」（不懂命理能否秒懂）
  分级：A 保留 / B 改写 / C 删除，改写清单给你过目后执行。
